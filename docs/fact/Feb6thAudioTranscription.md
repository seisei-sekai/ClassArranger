排课软件管理员说明（整理稿）

术语约定：文中“血管”统一为“学管”。

一、目标与核心原则

本系统用于替代当前人工排课流程，目标是：

让学管在提交约课时少填、少错、可追踪；

让教务在“分配负责—联系老师—占教室—确认学生—落地课表—录入系统—结算统计”全流程中可视化；

用系统规则约束高频错误（课时类型不匹配、时长不合规、教室冲突、跨校区不可达、排课超量等）；

最终做到：系统内数据唯一来源，避免“多个表并行导致不一致”。

二、账号与基础信息规则
1）登录后固定字段

学管提交约课时，以下信息应默认/固定，不允许随意改：

校区（来自账号）

学管姓名（来自账号）

若系统已分配账号，上述两项应“定死”。

2）学生信息检索

学管分配学生后，录入时可直接搜索学生姓名。

搜索结果应自动展示：

学生基础信息

可用课时（ERP、限定、班课兑换1v1等）

相关批次信息

能自动带出的字段不要求学管重复填写。

3）录入日期

约课提交日期默认“今天（提交当天）”，自动写入。

三、约课单字段与强约束
1）上课类型与课时

学管选择上课类型：

单次课 / 多次课（短期课 / 长期课）

时长可选：

1小时 / 2小时 / 3小时

强规则（必须限制）

1小时仅允许用于：

志望理由书纯修改（按你口述：“志望理由书纯修改”）

面试练习

其他内容必须 ≥ 2小时，不能由学管自由填写绕过。

若有特殊情况，需走“额外申请—管理员审批”流程（系统内审批节点）。

2）课时来源与可用性

需支持课时来源及剩余量展示：

国内课时

ERP

限定

班课兑换1v1

面试特训等

规则

学管可看到各类课时剩余（如 ERP 还剩多少、限定还剩多少）。

可提供“全用”快捷选项（如可用课时全部抵扣）。

班课兑换1v1相关逻辑需特殊控制（涉及退费风险）：

一对一可优先消耗（避免退费风险）

班课相关不可随意退（涉及国内顾问收入）

面试特训课时用途限制：

仅可用于按理由书/面试相关，不可用于不匹配科目（例如“日语”类错误使用）。

3）院校/学科字段触发条件

“志愿大学 / 学科”仅在特定业务场景必填，例如：

大学面试

按理由书

校内考（如日语、英语等）

其他如文书类可不填。

4）时间范围与可排截止日
起始时间选择方式

建议用日历范围选择（如起始到截止）。

管理员全局限制

管理员可设置“最迟可排到日期”（例如最迟 3月31日）。

超过该日期，学管不可继续提交（例如禁止排到 4月1日以后）。

5）学生希望时间段

支持两种录入方式并可融合：

选项型（早上/下午/晚上、尽量不周末等）

日历点选可上课日期 + 具体时间文本输入（如 10:00–12:00）

建议：

常见偏好给选项（减少自由输入混乱）

保留“其他备注”手动补充

最终由系统统一用于匹配

6）频次字段逻辑（长期/短期差异）

仅当选择“长期课”时，显示“频次”字段；

短期课一般不填频次，若有节奏要求（如隔周）写备注。

长期频次选项：

一周1次 / 2次 / 3次 / 4次 / 5次 ... 最多7次

上限强约束：每周最多7次

7）内容与备注

课上内容：必填

内容文本不做预限制（可自由发挥）

备注可自由填写

新增字段：指定老师（可选），有则填写老师名

8）总节数与分配方式

当为“多次课”时，需支持两种方式：

学管手动写总节数并分配节奏；

系统根据规则自动建议分配（例如按截止日期与频次平均分布），学管确认后生效。

四、教务“负责领取”与任务池机制

当前不适合“一键全部排完”，实际是分批处理（如300条中先处理一部分）。

系统需支持：

约课单列表中勾选多条；

点击“由我负责”后，这些条目标记当前教务姓名；

未领取条目保留在“未分配”池；

学管可见负责教务是谁（未领取则显示空）。

五、老师匹配规则（自动+人工）

系统匹配优先级建议：

专职老师优先（你口述“政策员”，应为“专职/正职老师”语义）

专职无空档再匹配兼职

老师优先级可由教务维护（后台可配）

六、教室与排课状态：必须区分“暂占”与“确认”

你提到非常关键的一点：

一键排完后，实际上还要经过老师/学管/学生多轮确认；

所以教室状态需要两级：

暂时占用（待确认）

最终确认（锁定）

防止：

还没确认就被别人占走；

或误以为已最终排定。

七、校区通行与连续课约束（关键业务规则）
1）跨校区可达性约束

如果老师 14:00–16:00 在旗舰校区上课，
则 16:00–18:00 不能直接排到板桥（无通行时间）。

系统应：

自动计算校区切换缓冲时间（至少1小时或按可配置规则）；

不满足通行条件时，禁排或提醒。

2）老师校区偏好

老师可维护：

可授课时段（如14:00–21:00）

可接受校区（旗舰 / 板桥 / 都可）

是否接受中途换校区

系统匹配时按偏好和现实可达性共同计算。

八、休息日/停课日管理

管理员可配置私塾休息日区间（例如 2月14日–18日）：

线下课：休息日禁止排课

线上课：可排（按规则放行）

日历上要清晰标记“休息”。

九、长期课表与学生端展示

目前人工做法是“底板复制+手动改色+逐日确认教室”，工作量大。

系统应支持：

自动生成学生课表（按确认结果）

学生端按日期聚合展示（只看自己相关时段，避免信息噪音）

校区+教室名称清晰显示（如“旗舰-个别指导室6”）

异动自动同步（取消、补课、换时段）

十、变更处理（高频）

现实中最常见：

学生临时改“晚上→早上”

某天不能上，要求换到其他天

老师临时冲突，需替换时段/替换老师

学管坚持“每周两次必须满足”

系统需内置变更流程：

发起变更（谁发起、原因）

重新匹配老师/教室

学管与学生确认

生效并写入日志（留痕）

十一、排课完成后的“落地录入”自动化

你提到当前要手动录入多个系统，且字段重复输入。

建议系统自动产出以下数据：

ERP编码

讲师

课程类型/科目

日期时间

校区教室

时长

录入人

并在提交前自动校验：

时长是否符合课种规则

老师是否撞课

教室是否撞课

课时是否超扣

十二、结算统计（月度）

你们当前要把每月课次再抄到“月度结算表”给老师核对时长，这是痛点。

系统需提供：

按月导出老师课时明细

按老师汇总总时长/总节数

支持老师确认（或导出核对单）

关键是“按时长结算”，时长字段必须准确可靠。

十三、群管理与通知模板自动化

当前人工流程：

建群

拉人（学管、领导、讲师、学生）

发群公告

发课表模板消息并@相关人

系统可做：

一键生成标准通知文案（自动带学生名、老师名、校区、时间）

模板可配置

状态联动（待确认/已确认/已变更）

十四、任务进度勾选与防漏

你现在靠“逐项打勾”避免漏步骤，这是正确需求。

系统需提供标准步骤流（可勾选）：

讲师时间确认

教室确认

课表完成

学管确认

学生确认

系统录入完成

群通知完成

并支持按“未完成步骤”筛选。

十五、实施前提（数据切换）

你明确提到关键前提：

若要启用新系统，先统一录入所有学生课时基线（每人一遍）；

启用后全部排课都走新系统；

禁止旧表/新系统并行长期运行，否则数据必然失衡。
